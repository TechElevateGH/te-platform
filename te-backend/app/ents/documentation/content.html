Platform Overview

TechElevate is a comprehensive talent enablement platform designed to streamline the job search and placement process. The platform serves as a centralized operating system connecting job seekers (members), company employees who can provide referrals (referrers), and TechElevate staff (leads and admins).

The platform facilitates the entire job hunting lifecycle - from resume preparation and application tracking to internal referral requests and skill development through curated learning content.

Codebase Map

The repository is split into two first-class applications with shared tooling at the root. Understanding the layout speeds up code reviews and onboarding.

Root / Tooling

  • Makefile: one-liner dev workflows such as make dev, make frontend-dev, make clean, and Docker helpers.
  • render.yaml: production backend definition (Python runtime, build.sh, start.sh, /health probe, and declared environment variables).
  • scripts/format.sh: battle-tested Ruff + isort formatter invoked by CI and make format.

te-backend/

  • app/core: settings, security helpers, and role-based permissions (permissions.py).
  • app/ents: feature modules (auth, users, resumes, applications, referrals, learning, documentation, etc.) with colocated CRUD + schemas.
  • app/database: Mongo client factory and startup scripts (see init_db.py for index creation).
  • docs/: Sphinx sources plus the built HTML mirrored into app/ents/documentation/content.html to keep the in-app docs evergreen.

te-frontend/

  • src/pages: route-level screens (e.g., Applications.jsx, Documentation.jsx, ResumeReviews.jsx, admin consoles, etc.).
  • src/components: feature widgets grouped by domain (learning, file uploads, referral flows, custom UI).
  • src/context: global stores (Auth, Data, Notification, Dark Mode) plus PostHog provider and hooks.
  • src/utils + src/hooks: shared Axios config, analytics helpers, and reusable React hooks.
Tip: When tracing a feature, start from app/ents/api.py to discover which router exposes the behavior, then jump into the matching frontend page to see how the data is consumed.

User Personas

Guest

Unauthenticated visitors to the platform with limited access.

Access: Public learning content only (view-only mode for lessons and practice problems)

Member

The primary user of the platform. Members are tech professionals or aspiring professionals looking for job opportunities. Stored in member_users collection.

Key Capabilities Members can upload and manage multiple versions of their resume (stored as embedded documents in resumes[] array), track job applications (stored in applications[] array), request referrals from company insiders, access all learning content with progress tracking, request resume reviews, manage referral essay and cover letter, and update their own profile. Requires email verification before first login.
Authentication: Email/password via /auth/login or Google OAuth via /auth/google

Referrer

Employees at tech companies who are willing to provide internal referrals to qualified members. Stored in privileged_users collection with company_id.

  • View referral requests filtered by their assigned company
  • Access resumes and essays from members
  • Approve or decline requests with feedback notes
  • Update referral status through the workflow
  • Track outcomes of provided referrals
Authentication: Token-only via /auth/referrer-login (matches password or lead_token field)

Volunteer

TechElevate volunteers who provide resume reviews and create learning content.

  • Review member resumes and provide detailed feedback
  • Create and publish learning lessons
  • Create other Volunteer accounts
  • All Member capabilities (can use platform as member)
Creation: Leads+ can create via POST /users/privileged/volunteers

Lead

TechElevate staff members who manage the platform's content and operations. Stored in privileged_users collection.

  • Curate and maintain company database via /referrals/companies
  • Access simplified company list for creating Referrers
  • View all member data and applications
  • Assign resume reviews to Volunteers
  • Create Volunteer and Referrer accounts
  • Manage all referral requests across companies
  • Oversee learning content and user progress
  • All lower role permissions
Authentication: Username/password via /auth/management-login

Admin

Platform administrators with full access to all system capabilities. Stored in privileged_users collection.

  • User management: Create Lead/Admin accounts via POST /users/privileged/leads
  • Delete any data (users, referrals, reviews)
  • System configuration and settings
  • Full analytics and reporting access
  • Content moderation across all entities
  • Override all permission checks
  • All lower role permissions
Authentication: Username/password via /auth/management-login
User Model Separation The platform uses two MongoDB collections for users:
  • member_users: Members with email-based auth, embedded applications/resumes, OAuth support
  • privileged_users: Referrers/Volunteers/Leads/Admins (roles 2-5) with username-based auth, company_id for Referrers, token authentication support

Domain Module Pattern

Each domain module (entity) follows a consistent structure:

  • models.py: Database models (Pydantic/MongoDB schemas)
  • schema.py: Request/response schemas for API
  • crud.py: Database operations (Create, Read, Update, Delete)
  • endpoints.py: FastAPI route handlers
  • dependencies.py: Dependency injection functions
Request Flow HTTP Request → FastAPI Router → Dependencies (auth, validation) → Endpoint Handler → CRUD Layer → Database → Response

Data Relationships

MongoDB uses embedded documents and references:

  • Embedded Documents: Applications and resumes are stored as arrays directly within member_users documents for fast access. Each has a UUID for identification.
  • Referenced Collections: Referrals reference user_id (member_users._id) and company_name (companies.name). Resume reviews reference user_id and optionally reviewed_by.
  • Denormalized Data: Company names and user information are cached in referrals and reviews for query performance. Locations are embedded subdocuments in companies.
  • Two User Collections: member_users and privileged_users are separate collections to optimize queries and enforce different validation rules.

Authentication & Authorization

The platform uses JWT (JSON Web Tokens) for stateless authentication with role-based access control. Three separate login endpoints serve different user types.

Authentication Flow

For Members:

  1. User submits email/password to POST /auth/login or uses Google OAuth
  2. Email verification required before first login (6-digit code sent via email)
  3. Backend validates credentials against member_users collection
  4. If valid, JWT token is generated with user ID and role
  5. Frontend stores token in localStorage
  6. Token included in Authorization header for subsequent requests

For Volunteer/Lead/Admin (roles 3-5):

  1. User submits username/password to POST /auth/management-login
  2. Backend validates against privileged_users collection with role >= 3
  3. Token generated and stored (same flow as Members)

For Referrers:

  1. User submits token only to POST /auth/referrer-login
  2. Backend matches token against privileged_users.password or privileged_users.lead_token
  3. Token generated with company_id for scoped access

Roles & Permissions

Role Value Permissions Login Endpoint
Guest 0 View public learning content only No auth required
Member 1 Submit applications/referrals, upload resumes, request reviews, access all learning content, manage own profile /auth/login or /auth/google
Referrer 2 View referral requests for assigned company, approve/decline requests, provide feedback /auth/referrer-login
Volunteer 3 Review resumes, create learning content, create other Volunteers, all Member permissions /auth/management-login
Lead 4 Manage companies, view all member data, create Volunteers/Referrers, assign resume reviews, all lower permissions /auth/management-login
Admin 5 Full system access, create any user type including Leads/Admins, delete data, system configuration /auth/management-login
Permission Hierarchy: Higher roles inherit all permissions from lower roles. The platform enforces this via require_member(), require_volunteer(), require_lead(), and require_admin() dependency functions.

Password Reset (Planned)

End-to-end password reset is rolling out next. The flow will reuse the verification service—users request a reset link, verify a short-lived token, and set a new password. Once complete, the endpoints documented below will activate.

  1. POST /auth/password-reset/request — accepts an email and sends a 6-digit code + secure token via email.
  2. POST /auth/password-reset/verify — validates the code/token combination (marks it used once confirmed).
  3. POST /auth/password-reset/complete — updates the hashed password after verification.
Security commitments
  • Tokens expire in 15 minutes and are single-use.
  • Rate limiting prevents repeated reset requests.
  • Reset attempts reuse the same audit + attempt tracking logic as email verification.

Email Verification System

Required for new Member accounts before first login. Ensures email ownership and reduces spam accounts. Stored in verification_codes collection.

Verification Flow

  1. User registers via POST /users - account created with email_verified: false
  2. Request code via POST /verification/send-code - 6-digit code sent to email, expires in 15 minutes
  3. User receives email with verification code
  4. Submit code via POST /verification/verify-email - validates code and sets email_verified: true
  5. User can now login via POST /auth/login

Verification Code Model

EmailVerification Document
  • email: Email address being verified
  • code: 6-digit verification code (randomly generated)
  • verification_type: "registration" or "email_change"
  • user_id: Reference to user (for email change requests)
  • new_email: New email address (for email change verification)
  • created_at: Timestamp when code was generated
  • expires_at: Expiration time (15 minutes from creation)
  • is_used: Boolean indicating if code has been successfully used
  • attempts: Number of failed verification attempts (max 5)

Email Change Flow

For existing members changing their email address:

  1. Request change via POST /verification/send-code-for-email-change with new email
  2. Receive code at new email address
  3. Verify code via POST /verification/verify-email-change
  4. Email updated in member_users collection

Verification API Endpoints

  • POST /verification/send-code - Send verification code to email (registration)
  • POST /verification/verify-email - Verify email with 6-digit code
  • POST /verification/resend-code - Resend code if expired or lost
  • POST /verification/send-code-for-email-change - Request email change (Member+)
  • POST /verification/verify-email-change - Complete email change (Member+)

Security Features

  • Rate limiting: Max 5 code requests per email per hour
  • Attempt tracking: Max 5 failed verification attempts per code
  • Expiration: Codes expire after 15 minutes
  • One-time use: Codes marked as used after successful verification
  • Email validation: Valid email format required

Resume Reviews

Members can request structured feedback on any uploaded resume. All review requests, assignments, and volunteer workflows live inside the consolidated app/ents/resume module, ensuring the same schemas power uploads and review tasks.

Highlights

Single Source

CRUD helpers, Pydantic schemas, and routers for uploads and reviews share the same package. Legacy imports from app.ents.resume_review forward to the new module.

Role Aware

Members submit requests, volunteers add feedback, and leads or admins manage assignments and bulk operations from one screen.

Assignment Tools

Leads can assign individual requests or bulk assign several to a volunteer, and reviewers see a dedicated “My Assignments” view.

Workflow

  1. Member submits a request with resume link, target role, and notes.
  2. Request enters the queue as Pending.
  3. Volunteers can self-update status to In Review when they begin, or leads can assign reviewers directly.
  4. Reviewer provides feedback and marks the request Completed when done.
  5. Members may cancel their own requests at any time before completion.

In Review Reviewer actively working on feedback. Assigned volunteer, lead, or admin
Completed Feedback delivered to the member. Assigned volunteer, lead, or admin
Cancelled Member withdrew the request before completion. Request owner

Referral System

Connect members with employees at target companies to request internal referrals. Managed through the referral entity (previously referral_company).

How It Works

For Members Browse companies via GET /referrals/companies where TechElevate has referrer connections. Submit a referral request with your resume and a personalized note. Track the status of your request via GET /referrals/mine and receive notifications when reviewed.
For Referrers View incoming referral requests for your company via GET /referrals/all (filtered by company_id). Download resumes, evaluate fit, and approve or decline with feedback via PATCH /referrals/{id}. Track outcomes of referrals provided.
For Leads/Admins Access simplified company list via GET /referrals/companies/list (id and name only) for dropdown selection when creating Referrer accounts. Create new referral-enabled companies via POST /referrals/companies.

Referral Request Flow

  1. Member discovers company with can_refer: true via companies endpoint
  2. Member submits request via POST /referrals with resume link, job details, and essay
  3. Referrer receives notification and views pending requests filtered by their company_id
  4. Referrer makes decision via PATCH /referrals/{id} with status and feedback
  5. Member receives notification of decision and feedback
  6. If approved, referrer submits internal referral to their company
  7. Member applies to job posting and tracks via Applications

Referral States

  • Pending - Awaiting referrer review (default state)
  • Under Review - Referrer is actively reviewing
  • Approved - Referrer will provide referral
  • Declined - Referrer declined with feedback in review_note
  • Submitted - Referral submitted to company HR
  • Completed - Process completed (hired or closed)

API Endpoints

  • GET /referrals/companies - List all referral-enabled companies with materials (All authenticated users)
  • GET /referrals/companies/list - Simplified company list for dropdowns (Lead+ only)
  • POST /referrals/companies - Create new company (Volunteer+ required)
  • GET /referrals/mine - Get current user's referral requests (Member+)
  • GET /referrals/all - Get all referrals (Lead+) or company-filtered (Referrer)
  • POST /referrals - Create referral request (Member+)
  • GET /referrals/{id} - Get specific referral (Owner or Lead+)
  • PATCH /referrals/{id} - Update referral status/feedback (Referrer for own company, Lead+ for all)
  • DELETE /referrals/{id} - Delete referral (Admin only)

Learning Modules

Curated learning content to help members improve technical skills and interview preparation. Content stored in lessons collection with progress tracked in user_progress.

Content Structure

Lesson Model Each lesson has:
  • title: Lesson name
  • category: Main category (e.g., "Arrays & Strings", "Dynamic Programming")
  • topic: Specific topic within category (e.g., "Two Pointers", "0/1 Knapsack")
  • description: Lesson summary and learning objectives
  • video_id: YouTube video ID for video content
  • content_type: "video", "article", "interactive", or "mixed"
  • difficulty: "Beginner", "Easy", "Medium", "Hard", or "Advanced"
  • tags[]: Keywords for search/filtering (e.g., ["sliding-window", "two-pointers"])
  • resources[]: Additional links with titles
  • code_examples[]: Code snippets by language
  • duration_minutes: Estimated completion time

Project Structure Best Practices

  • Keep routers thin: Business logic belongs in CRUD layer or services
  • Use Pydantic models: For request/response validation
  • Dependency injection: For database sessions and auth
  • Error handling: Use custom exceptions from utilities/errors.py

Adding a New Endpoint

# 1. Define schema in ents/[module]/schema.py
class NewItemCreate(BaseModel):
    name: str
    description: str

# 2. Add CRUD operation in ents/[module]/crud.py
async def create_item(db, item: NewItemCreate):
    result = await db.items.insert_one(item.dict())
    return result.inserted_id

# 3. Create endpoint in ents/[module]/endpoints.py
@router.post("/items")
async def create_new_item(
    item: NewItemCreate,
    db = Depends(get_database),
    current_user = Depends(get_current_user)
):
    item_id = await crud.create_item(db, item)
    return {"id": str(item_id), "message": "Item created"}

# 4. Register router in ents/api.py
from .module import endpoints as module_endpoints
api_router.include_router(module_endpoints.router)

Documentation Delivery

The documentation you are reading lives both inside the backend and on the static /documentation route inside the React app. Keeping it fresh requires two quick steps whenever content changes.

  1. Update source: edit app/ents/documentation/content.html (or regenerate from te-backend/docs via Sphinx) and ensure styles / scripts match the desired experience.
  2. Ship to frontend: the React Documentation.jsx page fetches this HTML via GET /v1/documentation, caches it in localStorage, and hydrates scripts/inline styles at runtime. Mobile users see cached copies if they’re offline.
Performance tips
  • GZip is enabled server-side for the HTML payload (GZipMiddleware in app/main.py).
  • The frontend automatically re-fetches in the background if the cache is older than 12 hours or an error occurs.
  • Use the “Open full page” fallback button to debug the raw HTML response if the embedded view misbehaves.

Security

Authentication Security

  • Password Hashing: bcrypt with salt rounds
  • JWT Tokens: Short expiration (7 days), secure secret key
  • Token Storage: httpOnly cookies or secure localStorage
  • Session Management: Automatic logout on token expiration

API Security

CORS Configuration

Restrict origins to known domains:

ALLOWED_ORIGINS = [
  "https://techelevate.org",
  "http://localhost:3000"
]

Rate Limiting

Prevent abuse with request limits:

  • 100 requests/minute per IP
  • 1000 requests/hour per user
  • Stricter limits on auth endpoints

Input Validation

All inputs validated via Pydantic:

  • Type checking
  • Length constraints
  • Regex patterns
  • SQL injection prevention

Data Security

  • Encryption at Rest: MongoDB Atlas encryption enabled
  • Encryption in Transit: TLS/SSL for all connections
  • Sensitive Data: Never log passwords or tokens
  • File Access: Drive files require authentication
  • Environment Variables: Secrets never in code

Security Checklist

  • ✅ Rotate service account keys every 90 days

  • ✅ Restrict MongoDB IP access whitelist
  • ✅ Audit role-based access controls
  • ✅ Enable JWT refresh tokens
  • ✅ Regular dependency security updates
  • ✅ Penetration testing quarterly
  • ✅ Security headers configured

Troubleshooting

Common Issues

Backend Won't Start

Problem ModuleNotFoundError or import errors
  • Verify virtual environment is activated
  • Reinstall dependencies: pip install -r requirements.txt
  • Check Python version: python --version (must be 3.9+)

MongoDB Connection Failed

Problem ServerSelectionTimeoutError
  • Verify MONGODB_URL in .env is correct
  • Check network connectivity to MongoDB Atlas
  • Ensure IP address is whitelisted in Atlas
  • Verify database user credentials

Google Drive Upload Fails

Problem 403 Forbidden or authentication errors
  • Verify GOOGLE_PRIVATE_KEY is properly formatted (with \n)
  • Check service account has domain-wide delegation
  • Ensure GOOGLE_FOLDER_ID exists and is accessible
  • Verify Drive API is enabled in Google Cloud Console

JWT Token Errors

Problem Could not validate credentials
  • Check SECRET_KEY matches between token creation and validation
  • Verify token hasn't expired (check exp claim)
  • Ensure Authorization header format: Bearer <token>
  • Clear browser localStorage and re-login

CORS Errors in Frontend

Problem Access to XMLHttpRequest blocked by CORS policy
  • Verify FRONTEND_URL in backend .env matches frontend origin
  • Check CORS middleware configuration in main.py
  • Ensure backend is running on correct port
  • Check browser console for actual error details

Frontend Build Fails

Problem npm build errors
  • Delete node_modules and package-lock.json
  • Run npm install fresh
  • Check for syntax errors in recent changes
  • Verify Node version compatibility
